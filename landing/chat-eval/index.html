Chat eval (meval)

End-to-end chat QA against the live Chat API

This page is the hub entry point. Today, runs are started from the repo (script below). Results can be written to Postgres when the DB URL is configured (not --dry-run).

Run the suite

From the Mobius repo root (Chat API and worker should be up):

./meval

The wrapper sets PYTHONPATH, loads mobius-chat/.env, and tees logs under mobius-qa/mobius-chat-qa/reports/.

Suite & questions

  • Canonical file: mobius-qa/mobius-chat-qa/eval_suite.yaml — edit in git to add, change, or retire cases.
  • Alternate file: python mobius-qa/mobius-chat-qa/run_eval.py --suite /path/to/suite.yaml (or pass the same flag through any wrapper you use).
  • Subset: --tests T01,T23 (comma-separated IDs).

Analytics & production

  • Schema: mobius-chat/db/schema/021_eval_tables.sql — tables eval_runs, eval_results.
  • Query those from your BI tool or add admin-only HTTP endpoints on Chat when you want a built-in dashboard.
  • For production, typical pattern is a scheduled job (or CI) that runs the same run_eval.py against the prod Chat base URL with secrets injected — not the dev-only Module Hub process.

Roadmap (if you want “click Run” here)

A “Run” button needs a small backend (async job, auth, timeouts) — comparable effort to wiring Retrieval Eval’s API. Keeping questions in YAML + git is the lightest path; a full in-browser editor + versioned DB suites is a larger product slice.
